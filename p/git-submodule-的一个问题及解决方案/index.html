Git Submodule 的一个问题及解决方案

Jul 30, 2024
阅读时长: 2 分钟

最近买了一台Windows笔记本，把原来写的博客从github上clone下来，再用原来的deploy脚本部署时发现会报这个错

1

fatal: in unpopulated submodule 'public'

其中public是hugo生成的内容的目录，检查才发现原来是因clone主项目的时候是直接用的git clone，而没有加submodule相关的命令，在这里解决问题只需要把public删除，再重新初始化一下submodule即可。

1
2

del .\public
git submodule update --init

这样git就会自动从远程下载public目录并正确地放在submodule中了。其他submodule相关的操作可以继续往下看。

这个错误通常出现在 Git 仓库中使用子模块（submodule）的情况下。出现这个错误的原因可能有以下几种:

  1. 子模块 ‘public’ 还没有被初始化或克隆:

    • 确保在主仓库中运行 git submodule init 和 git submodule update 来初始化和更新子模块。

  2. 子模块 ‘public’ 的 URL 配置不正确:

    • 检查主仓库中 .gitmodules 文件中 ‘public’ 子模块的 URL 是否正确。
    • 如果 URL 不正确，更新 URL 并运行 git submodule sync 来同步配置。

  3. 子模块 ‘public’ 的工作目录为空:

    • 进入子模块目录 public，检查是否有任何文件或提交记录。
    • 如果目录为空，可能需要重新克隆或初始化该子模块。

  4. 子模块 ‘public’ 已经被删除:

    • 检查主仓库的 .git/modules/ 目录下是否还有 ‘public’ 子模块的目录。
    • 如果目录不存在，可能需要从主仓库中移除该子模块的引用。

要解决这个问题,你可以尝试以下步骤:

  1. 确保 ‘public’ 子模块已经被正确初始化和更新:

    • git submodule init
    • git submodule update

  2. 检查 .gitmodules 文件中 ‘public’ 子模块的 URL 是否正确,如果不正确请更新 URL。

  3. 如果上述步骤无法解决问题,尝试从主仓库中删除 ‘public’ 子模块的引用,然后重新添加子模块:

    • git submodule deinit -f public
    • git rm -f public
    • 重新添加子模块 git submodule add <url> public
